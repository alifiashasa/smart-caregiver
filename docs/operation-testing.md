## **Glosarium Kolom**

1. **Jenis Case (+/-/Edge):**
   - **Positif (+):** Menguji alur normal (jalur sukses / happy path).
   - **Negatif (-):** Sengaja memasukkan data salah, akses ilegal, atau kondisi error untuk memastikan sistem menolak dengan baik.
   - **Edge:** Menguji nilai batas validasi (misal: tepat di batas min/max, data kosong, input 0 record, string maksimum).

2. **Precondition:** Kondisi yang harus terpenuhi _sebelum_ langkah eksekusi dilakukan (misal: "Caregiver sudah login dan punya token JWT", "Profil lansia sudah dibuat").

3. **Langkah Eksekusi:** Urutan tindakan teknis berupa HTTP request ke endpoint API. Gunakan **Postman**, **cURL**, atau **Thunder Client** untuk mengeksekusi.

4. **Data Uji:** Nilai spesifik yang dimasukkan dalam request body atau query parameter. Jika data berupa UUID, gunakan UUID yang didapat dari response endpoint sebelumnya.

5. **Expected Result:** Respon sistem yang diharapkan — mencakup HTTP status code, struktur response body, dan efek samping (notifikasi terkirim, data tersimpan di DB, alarm terbuat, dll).

6. **Status (OK/NG):** Diisi oleh tester saat eksekusi. `OK` jika sesuai expected. `NG` jika tidak sesuai → wajib lampirkan **Bug Report** dengan detail: endpoint, request/response aktual, ekspektasi, dan screenshot.

## **Tips untuk Praktikum di Kelas**

- **Jangan Hanya Jalur Hijau:** QA hebat mampu "merusak" aplikasi. Pastikan **≥30% test case adalah Negatif Case**. Dokumen ini memiliki **26 negatif** dari total **84 test case (~31%)**.

- **Status NG → Bug Report Wajib:** Jika status `NG`, catat di kolom _Remark_ atau lampirkan form Bug Report terpisah berisi:
  - Endpoint yang diuji
  - Request body & header yang dikirim
  - Response aktual vs response yang diharapkan
  - Screenshot hasil pengujian

- **Sinkronisasi PRD & BDD:** Setiap skenario di sini merujuk ke fitur di [PRD](prd.md) dan [BDD](bdd.md). Cross-check untuk memastikan tidak ada fitur yang terlewat.

- **Gunakan Environment Variables:** Simpan `BASE_URL`, `ACCESS_TOKEN`, `REFRESH_TOKEN`, dan `INTERNAL_API_KEY` di environment Postman agar mudah digunakan ulang di seluruh test case.

- **Urutan Eksekusi Disarankan:** Ikuti urutan ID test (`OP-01` → `OP-84`) karena banyak test case bergantung pada data yang dibuat di test case sebelumnya (dependency chain).

- **Tools Rekomendasi:** Postman / Thunder Client / cURL untuk manual testing. Newman (Postman CLI) untuk otomatisasi. Pytest untuk backend testing.

## **Catatan & Asumsi**

1. **Environment:** Seluruh pengujian dilakukan terhadap **local development server** (`http://localhost:8000`) dengan database PostgreSQL lokal atau SQLite (test config).

2. **OTP di Development:** Karena email OTP tidak benar-benar dikirim di environment lokal, tester bisa mengambil kode OTP dari log server atau membaca langsung dari database tabel OTP.

3. **Rate Limiter:** Test case `OP-06` mengandalkan rate limiter `5/minute` pada endpoint `/auth/register`. Jika di environment test rate limiter dimatikan, skip atau tandai sebagai `N/A`.

4. **LLM/AI untuk Rekomendasi:** Test case `OP-64` (generate rekomendasi AI) membutuhkan koneksi ke **Groq API** dengan API key valid. Jika API key tidak tersedia, test case ini bisa gagal — tandai sebagai `N/A` dan catat di Bug Report bahwa environment belum dikonfigurasi.

5. **Internal Jobs:** Endpoint `/internal/jobs/*` memerlukan `INTERNAL_API_KEY` yang dikirim via header `X-API-Key`. Pastikan key ini diatur di `.env` server.

6. **Google OAuth:** Fitur login Google OAuth (`/auth/google/*`) tidak dimasukkan dalam test case ini karena merupakan third-party integration yang memerlukan akun Google dan environment OAuth — dapat diuji terpisah sebagai _integration test_.

7. **Asumsi Akses:** Setiap caregiver hanya bisa mengakses data miliknya sendiri (`require_caregiver_owner`). Test case negatif akses lintas caregiver (OP-22, OP-66, OP-78) mengasumsikan sudah ada 2 akun caregiver terpisah di database.

8. **Mobile App Testing:** Dokumen ini fokus pada **Backend API testing**. UI testing untuk Flutter mobile app memerlukan dokumen terpisah dengan skenario yang mencakup navigasi, form UI, dan push notification — tetap bisa menggunakan test case yang sama sebagai dasar.
